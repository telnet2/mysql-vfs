## Architectural Overview

While the primary interface to the VFS is the main API service (`services/vfs`), the platform's reliability and asynchronous capabilities are handled by a set of independent microservices. These services operate in the background, using the shared MySQL database for coordination and task queuing.

This design decouples long-running or unreliable processes (like sending webhooks) from the main API, ensuring that the user-facing API remains fast and responsive.

The three core background services are:
1.  **Event Worker**: Processes internal events and creates webhook jobs.
2.  **Webhook Orchestrator**: Reliably delivers webhooks with retries and circuit breaking.
3.  **Scheduler**: Runs periodic maintenance tasks in a distributed-safe manner.

The data flow for an event is as follows:

```
1. VFS API
   - A file is created.
   - A record is inserted into the `events` table with status 'pending'.
   - API responds to user immediately.
     ↓
2. Event Worker
   - Polls the `events` table.
   - Finds the 'pending' event and marks it 'processing'.
   - Finds all webhooks subscribed to this event type.
   - Inserts records into the `webhook_jobs` table for each subscriber.
   - Marks the event 'completed'.
     ↓
3. Webhook Orchestrator
   - Polls the `webhook_jobs` table.
   - Finds pending jobs and sends the corresponding webhooks.
   - Manages retries, failures, and circuit breaking.
   - Marks the job 'acknowledged' or 'failed'.
```

## Event Worker

**Source Code:** `services/event-worker/main.go`

### Purpose

The Event Worker is the first stage in the asynchronous event processing pipeline. Its primary responsibility is to consume events generated by the main VFS service and prepare them for dispatch. It acts as a bridge between the synchronous creation of an event and the asynchronous, potentially unreliable, delivery of webhooks.

### Architecture & Logic

The service operates as a worker pool that polls the `events` table in the database.

1.  **Polling**: Workers periodically query the `events` table for records where `status = 'pending'` and the `visible_at` time has passed.
2.  **Claiming**: To prevent multiple workers from processing the same event, a worker "claims" an event by transactionally updating its status to `processing`.
3.  **Fan-out**: For each claimed event, the worker queries the `webhook_configs` table to find all active webhooks subscribed to that event's type. It then creates a new record in the `webhook_jobs` table for each subscriber. This is a "fan-out" process.
4.  **State Management**: After creating the necessary jobs, the event is marked as `completed`. If processing fails, the event's `retry_count` is incremented. After a maximum number of retries (`MaxRetries`), the event is moved to a dead-letter state (`status = 'dead_letter'`) for manual inspection.

This entire process is detailed in the `processEvent` function within `services/event-worker/main.go`.

### Configuration (Environment Variables)

| Variable | Description | Default |
|---|---|---|
| `DB_DSN` | The Data Source Name for the MySQL database connection. | `root:root@tcp(localhost:3306)/vfs?charset=utf8mb4&parseTime=True&loc=Local` |
| `WORKER_COUNT` | The number of concurrent workers to run in the pool. | `5` |
| `POLL_INTERVAL` | The time interval between polling the `events` table for new work. | `1s` |
| `BATCH_SIZE` | The maximum number of events to fetch in a single poll. | `10` |

## Webhook Orchestrator

**Source Code:** `services/webhook-orchestrator/main.go`

### Purpose

The Webhook Orchestrator is a robust and resilient service responsible for the actual delivery of webhooks to external endpoints. It is designed to handle network failures gracefully and to protect both itself and the receiving systems from being overwhelmed.

### Architecture & Logic

Like the Event Worker, this service runs a worker pool that polls the `webhook_jobs` table for pending jobs.

1.  **Job Polling**: Workers query for jobs where `status = 'pending'` and `next_retry_at` is in the past.
2.  **Webhook Dispatch**: For each job, the worker constructs an HTTP POST request containing the event payload.
3.  **Response Handling**: The worker evaluates the HTTP response. A status code in the `2xx` range is considered a success. Any other status code is treated as a failure, triggering the retry mechanism.

### Key Features

*   **Retry with Exponential Backoff**: When a webhook delivery fails, the job is not immediately discarded. Instead, it is rescheduled for a future time with an exponentially increasing delay. This prevents a failing endpoint from being hammered with constant retries. The logic is handled in the `processJob` function in `services/webhook-orchestrator/main.go`.
*   **Circuit Breaker Pattern**: To handle endpoints that are down for an extended period, the orchestrator implements a circuit breaker.
    *   If a webhook endpoint fails consecutively for a configured number of times (`CircuitBreakerThreshold`), the circuit is "opened," and the orchestrator will not attempt to send any webhooks to that URL for a cooldown period.
    *   After the cooldown, the circuit moves to a "half-open" state, where one test webhook is sent. If it succeeds, the circuit is "closed" and normal operation resumes. If it fails, the circuit re-opens for another cooldown period.
    *   This logic is implemented in the `recordFailure` and `recordSuccess` functions in `services/webhook-orchestrator/main.go`.
*   **HMAC Payload Signing**: To ensure security and allow receivers to verify the authenticity of a webhook, every outgoing request includes an `X-Signature` header. This signature is a `HMAC-SHA256` hash of the request body, signed with the webhook's configured secret. This is implemented in the `signPayload` function in `services/webhook-orchestrator/main.go`.

## Scheduler

**Source Code:** `services/scheduler/main.go`

### Purpose

The Scheduler is a distributed, database-backed cron job runner. It is responsible for executing periodic maintenance and cleanup tasks that are essential for the long-term health of the VFS platform.

### Architecture & Logic

The scheduler is designed to be run in a multi-instance configuration for high availability.

1.  **Distributed Locking**: To prevent a cron job from being executed by multiple scheduler instances at the same time, it uses the `cron_executions` table for leader election. When a job is due, all instances attempt to create a unique record for that job's specific execution time. The first instance to successfully create the record wins the "lease" and executes the job. This is handled in the `claimExecution` function in `services/scheduler/main.go`.
2.  **Lease Heartbeating**: The instance that holds the lease for a job periodically updates a `heartbeat_at` timestamp in the database.
3.  **Stale Lease Reaping**: A "reaper" process runs periodically to find jobs that have been running for too long without a heartbeat. This indicates that the scheduler instance holding the lease has died. The reaper marks these jobs as `recovered` so they can be investigated, preventing them from being stuck in a `running` state forever. This is implemented in the `reapStaleLeases` function in `services/scheduler/main.go`.

### Implemented Jobs

The scheduler comes with several built-in jobs, which are defined in the `executeHandler` function in `services/scheduler/main.go`:

*   **`cleanup_idempotency`**: This job purges expired records from the `idempotency_records` table. This is critical for preventing the table from growing indefinitely and confirms that the API's idempotency feature is stateful.
*   **`cleanup_events`**: This job deletes old, completed events from the `events` table, serving as a data retention policy to manage database size.
*   **`vacuum_s3`**: A placeholder for a future job intended to perform garbage collection in the S3 bucket by finding and deleting objects that are no longer referenced by any file version in the database.
